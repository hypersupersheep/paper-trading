# 协作交接

各 agent 之间**交接任务、汇报进度、共享记忆**的唯一公共文件。放在 app 仓库根,所有挂了本仓库的 agent 都能直接读写。

---

## 署名制度

- 每条记录**必须署名**:谁做的就署谁的代号,例:`— QR-A`。
- 格式:`### [日期] 标题 — 代号`,正文写「做了什么 / 给对方的需求 / 待确认」。
- 给别人派活/提需求,在条目里写清「**@对方代号:需要你做…**」;对方完成后**在同一文件追加回执并署名**。
- 不删别人的记录(append-only);要更正就追加一条说明。

### [2026-06-28] v1.15.3 真凶定位:zoneinfo 缺时区库崩溃(同事卡"启动本地引擎"的真因)— QR-A

v1.15.2 的启动崩溃落盘立功——同事 `startup_error.log` 给出真实报错:
`ModuleNotFoundError: No module named 'tzdata'` → `ZoneInfoNotFoundError: 'No time zone found with key Asia/Shanghai'`,崩在 `risk_store.py:14` `CN_TZ = ZoneInfo("Asia/Shanghai")`(模块加载期,`import backend.server` 直接失败)。

**根因**:`risk_store`/`scheduler_store` 用 `ZoneInfo("Asia/Shanghai")`,它需要系统时区库(`/usr/share/zoneinfo`)或打包进 `tzdata` 包。`zoneinfo` 惰性 import `tzdata`,PyInstaller 静态分析看不到 → **没打包**。我的开发机有系统时区库,所以**我的包在我机器上能跑**;但同事机器读不到系统时区库 → 崩。和 sleeve/迁移/绑定/防火墙全无关——前几轮全猜偏了,是 v1.15.2 的崩溃落盘才逼出真相。

**v1.15.3 双保险修复**:
1. `risk_store`/`scheduler_store` 把 `ZoneInfo("Asia/Shanghai")` 换成 `timezone(timedelta(hours=8))`(固定 UTC+8,与 `trading_store` 一致;中国大陆 1991 起恒 +8 无夏令时,语义等价)——**彻底不依赖系统时区库/tzdata**。
2. `paper_trading.spec` 把 `tzdata` 打进包(兜底 pandas/rqdatac 等运行时的 zoneinfo 用法)。

**实测**:断系统时区库环境(`PYTHONTZPATH=/nonexistent`)下,对照组 `ZoneInfo('Asia/Shanghai')` 复现同事的 `ZoneInfoNotFoundError`;修复后**打包二进制**仍正常起服务(200)、无报错。189 测试全绿,API 未变(v2)。

**给同事**:更新到 v1.15.3,这次是确定性修复,应能正常打开。
